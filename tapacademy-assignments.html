Rules of the game

apply to every project

R1Hand-written code

Semantic HTML and CSS written from scratch. No Bootstrap, no Tailwind, no copied templates. Vanilla JS only where a project demands it.

R2Responsive, mobile-first

Every page must work from 360px to desktop. Test in DevTools device mode before you call anything done.

R3Design with references

Study real sites on Refero.design, Godly and Land-book before you code. Rebuild at least one Aceternity UI or Magic UI component in plain CSS.

R4AI is allowed — disclosed

Use Antigravity, Claude or ChatGPT for scaffolding and debugging, but your README must state what AI did and what you wrote yourself. You must be able to explain any 10 lines of your CSS on demand.

R5One repo per project

Ten separate repos, each deployed on GitHub Pages, each with the live link in the repo description.

R6Clones stay honest

Clone projects use placeholder images (picsum.photos, Unsplash) and made-up content, and carry a “not affiliated” note in the README.

Ship it

repeat for every repo

Pushing code is not deploying. Run the commands, then flip the Pages switch — your site goes live at username.github.io/repo-name.

bash — inside your project folder 
# first-time setup for each project repo
$ git init
$ git add .
$ git commit -m "Initial commit"
$ git branch -M main
$ git remote add origin https://github.com/<username>/<repo-name>.git
$ git push -u origin main

# if GitHub created the repo with a README and push is rejected:
$ git pull origin main --rebase
$ git push -u origin main
Open repo → Settings → Pages. Under “Build and deployment”, set Source to Deploy from a branch.
Choose branch main and folder / (root), then hit Save. The build takes about a minute.
Copy your live URL and paste it into the repo description and your README. Then open it on your phone.

For projects 02 and 03: after deploying, add your GitHub Pages URL (e.g. https://<username>.github.io) as an allowed domain in your Clerk dashboard — otherwise Google login works on localhost and silently fails in production. Don’t lose an evening to this.

The README rule

non-negotiable · in all 10 repos

The README is the landing page of your code. Here's why it matters more than you think:

👀

Recruiters read it first

Nobody opens your CSS file. They open your repo, read the README for 30 seconds, and decide. A repo without a README looks abandoned — a great README looks professional.

🔎

Google & AI crawl GitHub

GitHub pages rank in Google search and feed AI assistants like ChatGPT, Perplexity & Gemini. Every README you publish is a mini SEO page for you — and for TAP Academy. That's GEO: Generative Engine Optimization, the 2026 evolution of SEO.

📈

Consistency compounds

When hundreds of TAP Academy students publish repos carrying the same rich TAP Academy section — placement in 60 days, AR classrooms, mock interviews — search engines start associating those keywords with the brand. Your repo becomes part of that story.

✅ Every README must include

  • Project name + one-line pitch — what is it, in one sentence
  • Live demo link — your GitHub Pages URL, right at the top
  • 1–2 screenshots — desktop + mobile view
  • Features list — the sections you built, as bullets
  • Tech stack — HTML · CSS · JS · AI-assisted features
  • Design references — Refero / Godly / Aceternity links you used
  • AI tools used — what you asked AI to build, honestly
  • What I learned — 2–3 lines in your own words
  • About TAP Academy section — keep it exactly as in the templates
  • Clone disclaimer — on projects 05 and 07–10, always
